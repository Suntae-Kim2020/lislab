1️⃣ DOI란 무엇인가?

📖 정의

DOI (Digital Object Identifier)는 디지털 객체를 영구적으로 식별하기 위한 국제 표준 식별자입니다.

쉽게 말하면, "인터넷 세상의 주민등록번호"입니다. 웹 주소(URL)는 바뀔 수 있지만, DOI는 절대 변하지 않습니다!

🏠 쉬운 비유: 집 주소 vs 주민등록번호

URL (집 주소)

🏠📍

예시: "서울시 강남구 테헤란로 123"

문제:

  • 이사 가면 주소가 바뀜
  • 건물이 철거되면 주소가 사라짐
  • 도로명이 변경되면 주소도 변경

DOI (주민등록번호)

👤🆔

예시: "123456-1234567"

장점:

  • 평생 변하지 않음
  • 어디로 이사 가도 동일
  • 이름이 바뀌어도 그대로

💡 핵심: URL은 "위치"를 가리키지만, DOI는 "대상 자체"를 가리킵니다!

2️⃣ DOI의 탄생 배경

🕰️ 1990년대 후반의 문제 상황

❌ URL의 심각한 문제

상황: 1998년, 한 연구자가 논문을 작성했습니다.

1998년

논문 출판, 참고문헌에 URL 기재

참고: http://old-journal.com/1998/vol5/paper123.pdf
2001년

저널이 새 출판사로 인수됨

→ 웹사이트 주소 변경
→ http://new-publisher.com/archives/...
2005년

또 다른 출판사로 재인수

→ 또 주소 변경
→ http://mega-publisher.com/legacy/...
2024년

원래 논문의 참고문헌을 확인하려고 함

http://old-journal.com/1998/vol5/paper123.pdf
→ 404 Not Found

😱 참고문헌 링크가 깨졌다!

📊 실제 통계 (연구 결과)

50%

5년 후 링크 깨짐률

75%

10년 후 링크 깨짐률

90%

20년 후 링크 깨짐률

→ 학술 연구의 연속성이 심각하게 위협받음!

✨ DOI의 탄생

1998년

국제 DOI 재단 (IDF) 설립

핵심 아이디어

"URL은 바뀌어도, DOI는 영원히 같은 자료를 가리킨다!"

→ 논문에 영구 식별자를 부여하자!

🔧 해결 방법

논문에 DOI 부여: 10.1234/journal.1998.123

1998년: DOI → http://old-journal.com/...
2001년: DOI → http://new-publisher.com/...  (URL 업데이트)
2005년: DOI → http://mega-publisher.com/... (URL 업데이트)
2024년: DOI → http://mega-publisher.com/... (여전히 작동!)

→ DOI는 변하지 않지만, 가리키는 URL은 업데이트됨!

3️⃣ DOI의 구조

🔍 DOI 해부하기

10.1234/journal.2024.v15.p123
10 DOI 시스템

모든 DOI는 "10"으로 시작

"이것은 DOI입니다"라는 표시

.
1234 출판사/기관 번호

DOI를 등록한 기관의 고유 번호

이 번호를 Prefix라고 부름

실제 예시:

  • 10.1038: Nature 출판사
  • 10.1016: Elsevier 출판사
  • 10.1109: IEEE
  • 10.5281: Zenodo (오픈 리포지토리)
/
journal.2024.v15.p123 개별 객체 식별 번호

해당 기관 내에서 부여한 고유 번호

이 번호를 Suffix라고 부름

자유로운 형식:

  • journal.2024.001
  • jlis-2024-v15-n2-p123
  • abc123xyz789
  • kim-2024-library-study

→ 기관이 알아서 정할 수 있음!

📋 DOI 구성 규칙

1. Prefix는 표준화

DOI 재단이 발급하는 공식 번호

예: 10.1234

2. Suffix는 자유

각 기관이 자유롭게 설계

예: abc123, journal.2024.001

3. 대소문자 구분 안 함

10.1234/ABC = 10.1234/abc

4. 영구 불변

한 번 부여되면 절대 변경 안 됨

CrossRef

https://www.crossref.org

설립: 2000년

전문 분야: 학술 논문, 저널 아티클, 학술 도서

주요 기능:

  • DOI 등록 및 관리
  • 참고문헌 연결 (Citation Linking)
  • 메타데이터 무료 공개
  • Similarity Check (표절 검사 도구)

회원: 전 세계 18,000+ 출판사 및 기관

실제 사례

Elsevier, Springer, Wiley, Taylor & Francis 등 거의 모든 주요 학술 출판사가 CrossRef를 통해 DOI를 등록합니다.

DataCite

https://datacite.org

설립: 2009년

전문 분야: 연구 데이터, 데이터셋, 소프트웨어, 회색문헌

주요 기능:

  • 연구 데이터에 DOI 부여
  • 데이터 인용 추적
  • FAIR 원칙 지원 (Findable, Accessible, Interoperable, Reusable)

회원: 전 세계 2,000+ 연구기관 및 리포지토리

실제 사례

PANGAEA (해양·지구과학 데이터), Zenodo (오픈 리포지토리), Figshare (연구 산출물) 등이 DataCite를 통해 데이터셋에 DOI를 부여합니다.

Korea DOI Center (KISTI)

https://www.doi.or.kr

운영 기관: 한국과학기술정보연구원 (KISTI)

역할: DataCite의 한국 대표 기관

주요 서비스:

  • 국내 기관에 DOI Prefix 발급
  • DOI 등록 및 관리 시스템 제공
  • DOI 사용 교육 및 컨설팅
  • 국내 연구 데이터 DOI 등록 지원
실제 사례

국내 대학 기관 리포지토리, 연구 데이터 센터, 정부 출연 연구소 등이 Korea DOI Center를 통해 DOI를 발급받고 있습니다.

예시 Prefix: 10.22800 (KISTI 자체)
국내 기관 DOI: 10.12345/data-2024-001

5️⃣ DOI 핸들 시스템

🔧 DOI의 핵심 기술: Handle System

Handle System은 DOI가 작동하는 기반 기술입니다. "주민등록번호로 현재 주소를 찾는 시스템"이라고 생각하면 됩니다.

🌐 DOI가 작동하는 과정

1
사용자가 DOI 클릭
https://doi.org/10.1234/article123
↓
2
Handle 서버에 질의

"10.1234/article123의 현재 URL은?"

↓
3
데이터베이스 조회
DOI: 10.1234/article123
→ URL: https://publisher.com/article/2024/123
   Updated: 2024-11-15
↓
4
사용자를 실제 URL로 리다이렉트
→ https://publisher.com/article/2024/123
↓
5
논문 페이지 표시

✅ 사용자가 원하는 논문을 봄

⚡ Handle System의 특징

🌍 분산 시스템

전 세계 여러 서버에 데이터 분산 저장

→ 한 곳이 망가져도 작동

⚡ 빠른 응답

밀리초 단위로 URL 조회

→ 사용자는 지연을 느끼지 못함

🔒 안정성

20년 이상 안정적으로 운영

→ 99.9% 이상 가용성

🔄 업데이트 가능

URL이 바뀌면 데이터베이스만 업데이트

→ DOI는 그대로 유지

⚠️ 중요: URL 업데이트의 중요성

핵심 원리: DOI가 작동하려면 현재 URL을 계속 업데이트해야 합니다!

❌ URL을 업데이트하지 않으면
2020년: DOI → http://old-site.com/article
2024년: 출판사가 웹사이트 변경
        BUT URL 업데이트 안 함!

→ DOI 클릭 시: http://old-site.com/article
→ 404 Not Found
→ DOI가 무용지물!
✅ URL을 제때 업데이트하면
2020년: DOI → http://old-site.com/article
2024년: 출판사가 웹사이트 변경
        → Handle DB 업데이트!
        DOI → http://new-site.com/articles/2020/123

→ DOI 클릭 시: http://new-site.com/articles/2020/123
→ ✅ 정상 작동!

💼 누가 업데이트하나요?

DOI를 등록한 출판사나 기관이 책임을 집니다.

  • 웹사이트 구조 변경 시
  • 다른 출판사로 이관 시
  • 서버 주소 변경 시

→ 반드시 DOI의 URL을 업데이트해야 합니다!

6️⃣ DOI 등록 과정

📝 기관이 DOI를 만드는 방법

STEP 1

Prefix 신청 및 발급

기관이 DOI 에이전시에 가입

# CrossRef, DataCite, 또는 Korea DOI Center에 신청

발급 예시:
→ Prefix: 10.1234

💰 연회비 납부 필요 (수십만~수백만 원)

STEP 2

DOI 생성

기관이 Suffix를 직접 설계

Prefix: 10.1234
Suffix 규칙 설계: journal.{year}.{vol}.{article}

생성된 DOI:
10.1234/journal.2024.v15.a001
10.1234/journal.2024.v15.a002
...
STEP 3

메타데이터 준비

DOI와 함께 제공해야 하는 정보

필수 메타데이터
{
  "DOI": "10.1234/journal.2024.v15.a001",
  "URL": "https://ourjournal.com/2024/vol15/article001",
  "title": "도서관 정보학의 미래",
  "authors": [
    {"name": "김도서", "orcid": "0000-0001-2345-6789"},
    {"name": "이정보", "orcid": "0000-0002-3456-7890"}
  ],
  "publication_date": "2024-03-15",
  "journal": "한국도서관정보학회지",
  "volume": "15",
  "issue": "2",
  "pages": "123-145",
  "abstract": "본 연구는...",
  "keywords": ["도서관", "정보학", "미래"],
  "references": [...]  // 참고문헌 목록
}

⚠️ URL은 필수입니다! DOI가 가리킬 주소를 반드시 지정해야 합니다.

STEP 4

에이전시에 등록

CrossRef, DataCite 등에 DOI + URL + 메타데이터 전송

등록 방법
  • 웹 인터페이스: 수동으로 입력
  • API: 프로그램으로 자동 등록
  • XML 파일 업로드: 대량 등록
# CrossRef API 예시
POST https://api.crossref.org/deposits
Content-Type: application/vnd.crossref.deposit+xml

<doi_batch>
  <head>...</head>
  <body>
    <journal>
      <journal_metadata>...</journal_metadata>
      <journal_article>
        <doi_data>
          <doi>10.1234/journal.2024.v15.a001</doi>
          <resource>https://ourjournal.com/2024/vol15/article001</resource>
        </doi_data>
      </journal_article>
    </journal>
  </body>
</doi_batch>
STEP 5

등록 완료 및 활성화

등록 후 몇 시간~1일 내 활성화

✅ 등록 완료!

이제 전 세계 어디서든:
https://doi.org/10.1234/journal.2024.v15.a001

→ 해당 논문 페이지로 이동됨!
지속 관리

URL 업데이트 (매우 중요!)

URL이 변경될 때마다 업데이트

# 웹사이트 변경 시
PUT https://api.crossref.org/works/10.1234/journal.2024.v15.a001
{
  "resource": "https://new-domain.com/articles/2024/15/001"
}

→ DOI는 그대로, URL만 업데이트!

⚠️ 이 단계를 빠뜨리면 DOI가 링크 깨진 URL을 가리키게 됩니다!

7️⃣ DOI 활용 사례

🔬 사례 1: 연구 데이터와 논문 연계

PANGAEA × Elsevier

배경

PANGAEA는 해양·지구과학 데이터를 보관하는 세계적인 데이터 리포지토리입니다. Elsevier는 과학 저널을 출판하는 대형 출판사입니다.

문제

연구자가 논문을 쓸 때 사용한 데이터를 어떻게 공유하고 인용할 것인가?

  • 논문: "우리는 2023년 남극 빙하 데이터를 분석했다"
  • 독자: "그 데이터를 어디서 볼 수 있나요?"
  • 기존 방식: 부록에 엑셀 파일 첨부 😢

✅ DOI 솔루션

1단계: 데이터 업로드

연구자가 PANGAEA에 데이터 업로드

데이터셋: "Antarctic Ice Sheet Measurements 2023"
파일: ice_data_2023.csv, metadata.xml
→ PANGAEA가 DOI 발급: 10.1594/PANGAEA.123456
2단계: 논문 작성

연구자가 Elsevier 저널에 논문 제출

논문 제목: "남극 빙하의 최근 변화 분석"
데이터 인용:
  Kim et al. (2023). Antarctic Ice Sheet Measurements 2023.
  PANGAEA. https://doi.org/10.1594/PANGAEA.123456

→ Elsevier가 논문에 DOI 발급: 10.1016/j.icarus.2024.001
3단계: 양방향 연결 
📄 논문 (10.1016/j.icarus.2024.001)
   → "이 논문에서 사용한 데이터: 10.1594/PANGAEA.123456"

📊 데이터 (10.1594/PANGAEA.123456)
   → "이 데이터를 사용한 논문: 10.1016/j.icarus.2024.001"
결과

✨ 완벽한 재현성!

  • 독자가 논문을 읽고 → 데이터 DOI 클릭 → 원본 데이터 다운로드
  • 데이터 재분석 가능
  • 데이터 제공자도 인용 실적 획득
  • 과학적 투명성 확보
📊 실제 효과

PANGAEA는 100만 개 이상의 데이터셋에 DOI를 부여했고, 이 데이터셋들은 수만 편의 논문에서 정확하게 인용되고 있습니다.

📋 사례 2: 메타데이터에서 DOI 활용

도서관 시스템의 서지 레코드

전자 저널 논문의 서지 정보를 관리할 때 DOI를 메타데이터 필드로 포함합니다.

MARC 레코드 예시
024 7_ $a 10.1234/jlis.2024.001 $2 doi
100 1_ $a 김도서, $d 1980-
245 10 $a 도서관 정보학의 미래 / $c 김도서
773 0_ $t 한국도서관정보학회지 $g Vol.15, No.2, p.123-145 (2024)
856 40 $u https://doi.org/10.1234/jlis.2024.001 $y 원문 보기
활용
  • 통합 검색: 여러 데이터베이스의 같은 논문을 DOI로 식별
  • 원문 연결: 이용자가 "원문 보기" 클릭 시 DOI로 리다이렉트
  • 중복 제거: 같은 DOI = 같은 논문으로 자동 판단
Dublin Core 메타데이터 예시
<metadata>
  <dc:title>도서관 정보학의 미래</dc:title>
  <dc:creator>김도서</dc:creator>
  <dc:date>2024-03-15</dc:date>
  <dc:identifier>https://doi.org/10.1234/jlis.2024.001</dc:identifier>
  <dc:relation>10.1594/PANGAEA.123456</dc:relation>
</metadata>

📚 사례 3: 참고문헌 자동 연결

CrossRef의 "참고문헌 연결" 서비스

작동 방식

1. 출판사가 논문의 참고문헌 목록을 CrossRef에 제공
2. CrossRef가 각 참고문헌의 DOI를 자동으로 찾음
3. 참고문헌에 DOI 링크 자동 생성
예시
논문 A의 참고문헌:
[1] Kim, D. (2023). Library Science. Journal of LIS, 15(2), 123-145.
    → CrossRef 자동 매칭: DOI 10.1234/jlis.2023.001 연결!

독자가 [1] 클릭 → 논문 원문으로 바로 이동!

✨ 효과: 학술 논문들이 서로 연결되어 "지식의 그물망" 형성

📖 2001년, DOI와의 첫 만남

2001년 KAIST 도서관에서 NDSL (National Discovery for Science Leaders) 시스템을 개발할 때의 실제 경험담입니다.

🚀 프로젝트 시작

2001년

목표: 국내외 과학기술 논문을 통합 검색하는 시스템 구축

과제: 전 세계 수백만 편의 논문을 수집하고, 원문으로 연결해야 함

당시 상황
  • DOI가 막 보급되기 시작한 시기
  • CrossRef도 이제 막 본격 가동 (2000년 설립)
  • 많은 출판사들이 DOI를 도입 중

💡 DOI LookUp 시스템 개발

아이디어: 논문의 DOI를 이용해 원문 링크를 자동으로 찾자!

시스템 구조
1. 출판사로부터 논문 메타데이터 수집
   → 제목, 저자, DOI 포함

2. NDSL 데이터베이스에 저장

3. 사용자가 논문 검색

4. DOI LookUp 시스템 작동:
   - 논문의 DOI를 CrossRef에 질의
   - CrossRef가 현재 원문 URL 반환
   - 사용자에게 "원문 보기" 링크 제공

😱 실시간 등록 문제 발생!

문제 상황

출판사로부터 수집한 메타데이터에 DOI가 있는데, CrossRef에 아직 등록되지 않은 경우가 빈번했습니다!

Day 1: 출판사가 논문 출판 + DOI 부여 
DOI: 10.1016/j.example.2001.001
Day 2: 출판사가 NDSL에 메타데이터 제공 
Title: "Example Paper"
DOI: 10.1016/j.example.2001.001
URL: https://publisher.com/article/2001/001
Day 3: NDSL에서 이 논문 검색됨
Day 3: 사용자가 "원문 보기" 클릭 
→ DOI LookUp 실행
→ CrossRef 질의: 10.1016/j.example.2001.001
→ CrossRef 응답: "NOT FOUND" 😱
Day 5~7: 출판사가 CrossRef에 등록 
→ 이제야 CrossRef에 등록됨!
→ 하지만 이미 이용자는 못 찾음...
왜 이런 일이?
  • 출판 속도 vs 등록 속도: 논문은 바로 출판되지만, CrossRef 등록은 며칠 걸림
  • 대량 일괄 등록: 출판사가 논문을 모아서 일주일에 한 번 등록
  • 등록 프로세스: 메타데이터 검증, 승인 과정이 필요

🔧 해결 방법

우리가 적용한 솔루션
1. Fallback URL 사용

CrossRef에서 DOI를 찾지 못하면, 출판사가 제공한 URL을 그대로 사용

try {
  url = crossref.lookup(doi);  // CrossRef 조회
} catch (NotFound) {
  url = metadata.original_url;  // 원본 URL 사용
}
2. 주기적 재확인

일주일 후 다시 CrossRef를 확인하여 URL 업데이트

// 매주 실행되는 배치 작업
for (article in articles_with_unresolved_doi) {
  if (crossref.has(article.doi)) {
    article.url = crossref.lookup(article.doi);
    article.url_source = "crossref";
  }
}
3. 사용자에게 안내

"DOI 조회 중..." 메시지 표시 후, 출판사 URL 제공

[원문 보기]
※ DOI 시스템에 등록 대기 중입니다.
  출판사 URL로 연결됩니다.
교훈

💡 DOI 시스템의 핵심:

  • DOI는 "즉시 등록"되는 게 아니라 "프로세스"를 거친다
  • 출판사와 DOI 에이전시 간의 동기화 시간차가 존재한다
  • 실시간 서비스를 만들 때는 Fallback 전략이 필수다

💡 가장 중요한 것

DOI는 단순한 "번호"가 아니라 "시스템"입니다.

DOI가 제대로 작동하려면 URL을 계속 업데이트해야 하며, 이는 출판사와 기관의 책임입니다.

DOI 덕분에 20년 전 논문도 지금 찾을 수 있고, 앞으로 50년 후에도 찾을 수 있을 것입니다!
