1️⃣ 식별자(Identifier)란 무엇인가?

📖 정의

식별자(Identifier)는 어떤 대상을 다른 것과 구별하여 유일하게 식별할 수 있게 해주는 이름이나 번호입니다.

🏫 일상생활 속 식별자

주민등록번호

대한민국 국민 한 명 한 명을 구별

123456-1234567

학번

학교에서 학생 개개인을 구별

2024001234

ISBN

전 세계 모든 책을 구별

978-89-1234-567-8

청구기호

도서관 내 각 책의 위치를 구별

020.1 김12ㄷ

🎯 좋은 식별자의 조건

1. 유일성 (Uniqueness)

같은 식별자를 가진 대상이 없어야 함

✅ 주민등록번호는 절대 중복되지 않음

2. 지속성 (Persistence)

시간이 지나도 변하지 않아야 함

✅ ISBN은 절판되어도 그대로 유지

3. 독립성

대상의 내용이나 위치가 변해도 식별자는 그대로

✅ DOI는 논문 URL이 바뀌어도 동일

2️⃣ URI란 무엇인가?

📖 정의

URI (Uniform Resource Identifier)는 인터넷 상의 자원을 식별하는 통일된 방식의 문자열입니다.

쉽게 말하면, "인터넷 세상의 모든 것을 가리키는 이름표"입니다.

🤔 쉽게 설명하면?

도서관 책 찾기로 비유해볼까요?

도서관에서 책을 찾으려면 두 가지 방법이 있습니다:

  1. 청구기호로 찾기: "020.1 김12ㄷ"

    → 이름만으로 구별. 책이 어디 있든 이 번호는 불변

  2. 서가 위치로 찾기: "3층 A서가 2번째 줄 5번째"

    → 위치를 알려줌. 실제로 가서 찾을 수 있음

URI는 이 두 가지를 모두 포함하는 개념입니다!

📊 URI의 구성

URI

통일된 자원 식별자

URL

위치로 찾기

https://library.com/book/123
URN

이름으로 찾기

urn:isbn:978-89-1234-567-8

💡 URI = URL + URN (URL과 URN을 모두 포함하는 상위 개념)

3️⃣ URL (Uniform Resource Locator)

📖 정의

URL은 자원의 위치(Location)를 나타내는 주소입니다.

"어떻게 가면 찾을 수 있는지" 위치 정보를 제공합니다.

🔍 URL의 구조

https://library.example.com:443/books/detail?id=123#chapter2
https://

프로토콜 (통신 방법)

library.example.com

도메인 (어느 서버?)

:443

포트 (출입구 번호)

/books/detail

경로 (서버 내 위치)

?id=123

쿼리 (추가 정보)

#chapter2

프래그먼트 (페이지 내 위치)

✅ URL의 장점

  • 직접 접근 가능: 웹브라우저에 입력하면 바로 해당 자원으로 이동
  • 사람이 읽기 쉬움: 주소만 봐도 어떤 내용인지 짐작 가능
  • 계층 구조: 폴더처럼 체계적으로 정리 가능

❌ URL의 단점

  • 위치 의존성: 서버를 옮기거나 폴더 구조를 바꾸면 URL이 변경됨
  • 링크 깨짐: 서버가 사라지면 URL이 작동하지 않음 (404 Not Found)
  • 중복 가능성: 같은 자원이 여러 URL을 가질 수 있음

💡 URL 실제 사례

웹사이트 주소

https://www.naver.com
https://www.google.com/search?q=도서관

도서관 검색 결과

https://library.example.com/search?keyword=도서관학&page=2

대학교 홈페이지

https://www.university.ac.kr/about/history.html

⚠️ URL의 문제점 사례

상황: 2010년에 작성한 논문에서 참고문헌으로 URL을 인용했습니다.

참고문헌: http://old-library-site.com/resource/article123.pdf

2024년 현재: 해당 도서관이 웹사이트를 새로 만들면서 주소가 변경됨

404 Not Found - 페이지를 찾을 수 없습니다

→ 논문의 참고문헌이 더 이상 유효하지 않게 됨!

→ 이 문제를 해결하기 위해 URN이 필요합니다.

4️⃣ URN (Uniform Resource Name)

📖 정의

URN은 자원의 이름(Name)으로 식별하는 방식입니다.

위치와 무관하게 영구적이고 유일한 이름을 제공합니다.

🤔 쉽게 설명하면?

📚 책을 찾는 두 가지 방법

URL 방식:

"3층 A서가 2번째 줄 5번째에 있는 빨간 표지 책"

→ 책을 다른 서가로 옮기면? 못 찾음!

URN 방식:

"ISBN 978-89-1234-567-8인 책"

→ 책이 어디로 옮겨가든 ISBN으로 찾을 수 있음!

🔍 URN의 구조

urn:isbn:978-89-1234-567-8
urn

URN임을 나타냄

isbn

네임스페이스 (어떤 체계?)

978-89-1234-567-8

고유 식별자

✅ URN의 장점

  • 영구성: 자원이 어디로 이동해도 이름은 변하지 않음
  • 위치 독립성: 서버가 바뀌어도 같은 URN으로 식별 가능
  • 전 세계적 유일성: 중복 없이 유일하게 식별
  • 신뢰성: 학술 논문 등에서 안전하게 참조 가능

❌ URN의 단점

  • 직접 접근 불가: URN만으로는 바로 자원에 접근할 수 없음
  • 해석기 필요: URN을 URL로 변환해주는 시스템이 필요함
  • 관리 비용: URN 체계를 유지하려면 중앙 관리 조직이 필요
  • 보급률 낮음: URL에 비해 아직 덜 사용됨

💡 URN 실제 사례

ISBN (국제표준도서번호)

urn:isbn:978-89-7050-123-4
→ 전 세계 어디서든 이 번호로 책을 식별

ISSN (국제표준연속간행물번호)

urn:issn:1234-5678
→ 학술지, 잡지 등을 식별

OID (객체 식별자)

urn:oid:2.16.840.1.113883.6.238
→ 의료 정보 시스템에서 사용

5️⃣ DOI: 학술 세계의 영구 식별자

📖 정의

DOI (Digital Object Identifier)는 디지털 객체를 영구적으로 식별하기 위한 국제 표준 식별자입니다.

주로 학술 논문, 연구 데이터, 학술 자료에 사용됩니다.

🎯 DOI는 URI의 어디에 해당할까?

DOI는 URN의 일종입니다!

DOI는 이름 기반 식별자로, URN의 특징을 모두 가지고 있습니다:

  • ✅ 영구적 (논문 URL이 바뀌어도 DOI는 불변)
  • ✅ 유일함 (전 세계에서 중복 없음)
  • ✅ 위치 독립적 (어느 저널로 이동해도 동일)

하지만 URL로도 사용 가능!

DOI는 특별하게도 URL로 변환하여 직접 접근할 수 있습니다:

DOI: 10.1000/182
→ URL: https://doi.org/10.1000/182

→ URN의 영구성 + URL의 접근성 모두 확보!

🔍 DOI의 구조

10.1234/jlis.2024.001
10

DOI 시스템 식별

1234

출판사/기관 번호

/

구분자

jlis.2024.001

개별 논문 고유번호

📌 DOI 사용 사례

사례 1: 학술 논문 인용

❌ URL로 인용한 경우
김도서 (2020). 도서관 정보학의 미래.
출처: http://old-journal.com/vol12/article5.pdf

→ 저널사가 웹사이트를 변경하면 링크가 깨짐

✅ DOI로 인용한 경우
김도서 (2020). 도서관 정보학의 미래.
DOI: https://doi.org/10.1234/jlis.2020.12.5

→ 웹사이트가 바뀌어도 DOI는 항상 논문을 찾아줌

사례 2: 연구 데이터 공유

상황: 연구자가 설문조사 데이터를 공개하고 싶습니다.

해결: 데이터에 DOI를 부여

데이터셋: "한국 대학도서관 이용자 만족도 조사 (2024)"
DOI: https://doi.org/10.5061/dryad.12345

효과:

  • ✅ 다른 연구자들이 정확하게 인용 가능
  • ✅ 데이터 저장소가 바뀌어도 DOI로 찾을 수 있음
  • ✅ 원 연구자에게 인용 실적으로 인정

사례 3: CrossRef 시스템

CrossRef는 세계 최대의 DOI 등록 기관입니다.

1단계

저널이 논문을 출판하면서 DOI 부여

DOI: 10.1234/jlis.2024.001
→
2단계

CrossRef에 메타데이터 등록
(제목, 저자, 초록 등)

→
3단계

DOI로 접근하면 현재 URL로 리다이렉트

https://doi.org/... → 실제 논문 페이지

→ 논문이 다른 출판사로 이전되어도 DOI는 그대로 작동!

📊 DOI vs URL 비교

특성 URL DOI
영구성 ❌ 웹사이트 변경 시 깨짐 ✅ 영구적으로 유지
직접 접근 ✅ 브라우저에 바로 입력 ✅ doi.org를 통해 접근
관리 ❌ 각 사이트가 개별 관리 ✅ 중앙 시스템(CrossRef)이 관리
인용 ❌ 학술 인용에 부적합 ✅ 학술 인용 표준
비용 ✅ 무료 ❌ 등록 비용 발생

6️⃣ 비교 정리: URI vs URL vs URN

구분 URI URL URN
의미 통일된 자원 식별자
(상위 개념)		 자원의 위치 자원의 이름
예시 모든 인터넷 주소 https://library.com/book/123 urn:isbn:978-89-1234-567-8
특징 URL과 URN을 포함 위치 기반, 직접 접근 이름 기반, 영구적
장점 - 사용하기 쉬움 변하지 않음
단점 - 링크가 깨질 수 있음 해석기가 필요함
도서관 비유 책을 찾는 모든 방법 서가 위치로 찾기 청구기호로 찾기

🎨 관계도

URI

모든 인터넷 식별자

URL

URN

💡 모든 URL과 URN은 URI입니다. 하지만 모든 URI가 URL이나 URN인 것은 아닙니다.

7️⃣ 도서관에서의 활용

📚 도서관 실무 적용 사례

1. 장서 관리

  • ISBN: 각 도서를 전 세계적으로 유일하게 식별
  • 청구기호: 도서관 내에서 자료의 위치를 식별
  • 바코드: 도서관 시스템 내에서 개별 책을 추적
URN 방식: urn:isbn:978-89-1234-567-8
도서관 내부 식별: BC20240001234
청구기호: 020.1 김12ㄷ c.2

2. 전자 자원 관리

  • DOI: 전자저널 논문을 영구적으로 식별
  • URL: 구독 데이터베이스에 접근
  • OpenURL: 도서관의 보유 자료로 연결
# 사용자가 논문 DOI를 클릭
DOI: 10.1234/jlis.2024.001

# 도서관 시스템이 확인
→ 우리 도서관이 이 저널을 구독 중인가?
→ YES → 구독 사이트의 전문 페이지로 이동
→ NO → 타 도서관 원문복사 서비스 안내

3. 디지털 아카이브

  • Handle System: 디지털 객체의 영구 식별자
  • ARK: 문화유산 자료의 지속가능한 식별자
# 국립중앙도서관 디지털 컬렉션
Handle: hdl.handle.net/20.500.12345/67890

# 대학 기관 리포지토리
http://repository.university.ac.kr/handle/123456789/12345
