1️⃣ REST API란 무엇인가?

📖 정의

REST (Representational State Transfer)는 웹의 장점을 최대한 활용할 수 있는 아키텍처 스타일입니다.

REST API는 REST 원칙을 따르는 웹 서비스 인터페이스로, 클라이언트와 서버 간의 데이터 교환을 표준화된 방식으로 수행합니다.

🍕 레스토랑 비유로 이해하기

레스토랑 시스템

  • 손님: 클라이언트 (웹 브라우저, 앱)
  • 메뉴판: API 문서
  • 주문: HTTP 요청 (Request)
  • 주방: 서버
  • 음식: 데이터 (JSON)
  • 서빙: HTTP 응답 (Response)

REST API 시스템

  • 클라이언트: 데이터를 요청하는 쪽
  • API 엔드포인트: 주문 가능한 메뉴
  • GET /pizzas: "피자 목록 주세요"
  • 서버: 요청을 처리하고 데이터 준비
  • JSON 데이터: 요청한 정보
  • 200 OK: "주문 완료했습니다"

🎯 REST의 핵심 개념

1. 자원 (Resource)

URI로 식별되는 모든 것

/users/123

→ ID가 123인 사용자

2. 행위 (Verb)

HTTP 메서드로 표현

GET, POST, PUT, DELETE

→ 조회, 생성, 수정, 삭제

3. 표현 (Representation)

자원의 상태를 나타내는 형식

JSON, XML

→ 주로 JSON 사용

2️⃣ REST의 6가지 원칙

1️⃣ 클라이언트-서버 구조 (Client-Server)

의미: 클라이언트와 서버가 독립적으로 진화할 수 있어야 합니다.

예시

✅ 좋은 예:

  • 서버: 데이터 저장 및 처리만 담당
  • 클라이언트: UI 표시만 담당
  • → 서버를 변경해도 클라이언트에 영향 없음

2️⃣ 무상태성 (Stateless)

의미: 각 요청은 독립적이며, 서버는 클라이언트의 상태를 저장하지 않습니다.

🤔 쉽게 설명하면?

서버는 "건망증"이 있습니다. 방금 전에 누가 왔는지 전혀 기억하지 못해요. 그래서 매번 "저 김도서입니다"라고 신분증을 보여줘야 합니다.

📖 도서관 비유

❌ 나쁜 예 (Stateful - 상태 기억):

첫 방문: "안녕하세요, 저는 학번 2024001 김도서입니다."

사서: "네, 기억할게요!" (메모장에 적음)

두 번째 방문: "책 대출할게요."

사서: "아, 김도서님이시죠? 네!" (메모장 확인)

⚠️ 문제: 사서가 메모장을 잃어버리면? 방문자가 너무 많으면 메모장이 꽉 찰 수 있어요!

✅ 좋은 예 (Stateless - 상태 저장 안 함):

첫 방문: "안녕하세요, 저는 학번 2024001 김도서입니다."

사서: "여기 회원카드 드릴게요!" (카드에 학번 적혀있음)

두 번째 방문: "책 대출할게요. (카드 제시)"

사서: "카드 확인... 2024001 김도서님이시군요. 네!"

✅ 장점: 사서는 아무것도 기억할 필요 없음. 카드만 확인하면 돼요!

💡 용어 설명
  • 세션 (Session): 서버의 "메모장". 방문자 정보를 임시로 저장하는 공간
  • 토큰 (Token): "회원카드". 사용자 정보가 담긴 암호화된 문자열
  • 무상태성: 서버가 메모장을 사용하지 않고, 매번 카드를 확인하는 방식

3️⃣ 캐시 가능 (Cacheable)

의미: 응답은 캐시 가능 여부를 명시해야 합니다.

🤔 캐시(Cache)가 뭐예요?

캐시는 "복사본 보관함"입니다. 자주 보는 정보를 미리 복사해서 가까운 곳에 보관해두면, 멀리 있는 원본을 매번 가져올 필요가 없어요!

📖 도서관 비유

캐시 없이:

학생: "도서관 이용 안내문 주세요."

사서: "잠깐만요. 서고에 가서 가져올게요..." (5분 소요)

다른 학생: "저도 도서관 이용 안내문 주세요."

사서: "또 서고에 가야 하네요..." (5분 소요)

⚠️ 비효율적! 같은 일을 반복하고 있어요.

캐시 사용:

학생: "도서관 이용 안내문 주세요."

사서: "서고에서 가져올게요... 자, 여기 있어요. 복사본도 책상에 놔둘게요!" (캐시 저장)

다른 학생: "저도 도서관 이용 안내문 주세요."

사서: "여기 책상에 있어요!" (캐시에서 바로 제공 - 즉시 제공)

✅ 효율적! 한 번 가져온 건 복사본을 활용해요.

💡 용어 설명
  • Cache-Control: "이 정보는 1시간 동안 복사본을 써도 돼요"라고 알려주는 표시
  • max-age=3600: 3600초 (1시간) 동안 캐시 사용 가능
  • ETag: "버전 번호". 원본이 바뀌었는지 확인하는 표시 (예: "v1", "v2")
왜 중요한가요?

서버와의 통신을 줄이면:

  • ⚡ 속도가 빨라집니다 (복사본은 내 컴퓨터에 있으니까!)
  • 💰 서버 부담이 줄어듭니다
  • 📱 데이터 사용량이 줄어듭니다

4️⃣ 계층화 시스템 (Layered System)

의미: 클라이언트는 직접 서버에 연결되었는지 중간 서버를 거쳤는지 알 수 없어야 합니다.

🤔 쉽게 설명하면?

피자를 주문할 때, 주방에서 직접 받는지 아니면 중간에 배달 대행업체를 거치는지 손님은 몰라도 돼요. 그냥 "피자 주문했어요, 피자 받았어요"만 알면 됩니다!

📖 도서관 비유

간단한 구조:

이용자 → (바로) → 사서

이용자가 사서에게 직접 요청

계층화된 구조:

이용자 → 안내 데스크 → 담당 부서 → 실제 담당 사서

1️⃣ 안내 데스크: "어느 부서로 안내할까요?"

2️⃣ 담당 부서: "지금 바쁘지 않은 사서에게 연결할게요"

3️⃣ 실제 담당 사서: 실제 업무 처리

✅ 이용자는 중간 단계를 몰라도 됨. 요청만 하면 결과를 받아요!

💡 용어 설명 (웹 서버의 중간 계층들)
  • Load Balancer (로드 밸런서): "업무 분배 담당자"

    여러 서버 중에서 지금 덜 바쁜 서버로 요청을 보내주는 역할. 마치 은행에서 "2번 창구로 가세요"라고 안내하는 것처럼!

  • API Gateway (API 게이트웨이): "입구 경비원"

    모든 요청을 처음 받아서 확인하는 곳. "신분증 확인하고, 어느 부서로 갈지 안내해드릴게요" 같은 역할!

  • 서버 (Server): "실제 일하는 사람"

    실제로 데이터를 처리하고 응답을 만드는 곳

왜 이렇게 복잡하게 만들까요?
  • 🛡️ 보안: 입구 경비원(Gateway)이 위험한 요청을 막아줘요
  • ⚖️ 부하 분산: 한 서버가 너무 바쁘면 다른 서버로 분산
  • 🔧 유연성: 중간 계층을 바꿔도 사용자는 모르고 계속 사용 가능
  • 📈 확장성: 서버를 더 추가해도 사용자는 똑같이 사용

5️⃣ 인터페이스 일관성 (Uniform Interface)

의미: 일관된 인터페이스로 시스템 아키텍처를 단순화합니다.

🤔 쉽게 설명하면?

모든 도서관이 똑같은 방식으로 운영된다고 생각해보세요.

  • 어느 도서관을 가든 청구기호로 책을 찾고
  • 대출증으로 책을 빌리고
  • 반납함에 책을 반납합니다

→ REST API도 마찬가지! 어떤 서비스든 같은 규칙으로 작동하니까 배우기 쉬워요.

4가지 제약 조건

  • 자원 식별: URI로 자원을 식별

    → 책의 청구기호처럼, 각 데이터에 고유한 주소가 있어요

  • 메시지를 통한 자원 조작: HTTP 메서드 사용

    → GET(조회), POST(생성), PUT(수정), DELETE(삭제) 같은 정해진 동작만 사용

  • 자기 서술적 메시지: 메시지만으로 이해 가능

    → 응답 자체에 "이건 JSON이고, 이렇게 읽으세요" 정보가 다 들어있어요

  • HATEOAS: 하이퍼링크로 다음 가능한 행동 안내

    → 웹페이지의 링크처럼, "다음엔 이걸 할 수 있어요" 라고 알려줘요

💡 HATEOAS 쉽게 이해하기

HATEOAS (Hypermedia As The Engine Of Application State)

웹 브라우저로 쇼핑몰을 생각해보세요:

  1. 상품 페이지를 보면 → "장바구니에 담기", "바로 구매하기" 링크가 보여요
  2. 장바구니에 담으면 → "계속 쇼핑하기", "결제하기" 링크가 보여요
  3. 결제하면 → "주문 내역 보기", "영수증 출력" 링크가 보여요

→ 다음에 뭘 할 수 있는지 링크로 알려주는 것이 바로 HATEOAS입니다!

예시: 도서 대출 API의 HATEOAS
{
  "bookId": 123,
  "title": "도서관학 개론",
  "status": "available",
  "_links": {
    "self": "/api/books/123",
    "borrow": "/api/books/123/borrow",     ← 대출 가능!
    "reserve": "/api/books/123/reserve"    ← 예약 가능!
  }
}

→ API 응답에 "다음 단계" 링크가 포함되어 있어서, 클라이언트는 이 링크만 따라가면 됩니다.

6️⃣ Code on Demand (선택사항)

의미: 서버가 클라이언트에 실행 가능한 코드를 전송할 수 있습니다.

예시

JavaScript 파일을 서버에서 다운로드하여 클라이언트에서 실행

3️⃣ HTTP 메서드: CRUD의 기본

REST API는 HTTP 메서드를 사용하여 자원에 대한 작업을 정의합니다.

CRUD: Create (생성), Read (조회), Update (수정), Delete (삭제)

💡 자주 나오는 용어 미리 알아두기

멱등성 (Idempotent)

같은 요청을 1번 보내든 100번 보내든 결과가 똑같은 성질을 말해요.

📚 도서관 비유:

  • 멱등성 O: "3층 A-123 서가의 책 목록을 보여주세요"

    → 몇 번을 물어봐도 같은 목록이 나와요 (GET)

  • 멱등성 O: "도서 ID 123번을 삭제해주세요"

    → 이미 삭제된 책을 다시 삭제해도 결과는 같아요 - "없음" (DELETE)

  • 멱등성 X: "새 도서를 등록해주세요"

    → 요청할 때마다 새 책이 계속 추가돼요! (POST)

왜 중요한가요?

네트워크가 불안정해서 같은 요청이 실수로 여러 번 전송될 수 있어요. 멱등성이 있으면 실수로 2번 보내도 안전하지만, POST처럼 멱등성이 없으면 같은 데이터가 중복으로 생성될 수 있습니다.

안전 (Safe)

서버의 데이터를 전혀 변경하지 않는 성질입니다.

  • 안전 O: GET (조회만 하니까)
  • 안전 X: POST, PUT, DELETE (데이터를 변경하니까)
GET

데이터 조회 (Read)

용도: 서버에서 데이터를 가져옵니다. 서버 상태를 변경하지 않습니다.

특징: 안전(Safe), 멱등성(Idempotent), 캐시 가능

실제 예시 1: 도서 목록 조회

📤 요청 (Request)
GET /api/books HTTP/1.1
Host: library.example.com
Accept: application/json
Authorization: Bearer abc123token
📥 응답 (Response)
HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: max-age=3600

{
  "books": [
    {
      "id": 1,
      "title": "도서관학 개론",
      "author": "김도서",
      "isbn": "978-89-1234-567-8",
      "status": "available"
    },
    {
      "id": 2,
      "title": "정보검색론",
      "author": "이정보",
      "isbn": "978-89-9876-543-2",
      "status": "checked_out"
    }
  ],
  "total": 2,
  "page": 1
}

실제 예시 2: 특정 도서 조회

📤 요청
GET /api/books/1 HTTP/1.1
Host: library.example.com
Accept: application/json
📥 응답
HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 1,
  "title": "도서관학 개론",
  "author": "김도서",
  "isbn": "978-89-1234-567-8",
  "publisher": "학문사",
  "published_date": "2024-01-15",
  "status": "available",
  "location": "3층 서고 A-123"
}

💡 주의: GET은 절대 데이터를 변경하면 안 됩니다!

❌ GET /api/books/1/delete ← 이렇게 하면 안 돼요!

✅ DELETE /api/books/1 ← DELETE 메서드를 사용하세요!

POST

데이터 생성 (Create)

용도: 서버에 새로운 데이터를 생성합니다.

특징: 안전하지 않음, 멱등성 없음

실제 예시: 새 도서 등록

📤 요청
POST /api/books HTTP/1.1
Host: library.example.com
Content-Type: application/json
Authorization: Bearer abc123token

{
  "title": "메타데이터의 이해",
  "author": "박메타",
  "isbn": "978-89-5555-123-4",
  "publisher": "정보출판사",
  "published_date": "2024-12-01"
}
📥 응답
HTTP/1.1 201 Created
Content-Type: application/json
Location: /api/books/3

{
  "id": 3,
  "title": "메타데이터의 이해",
  "author": "박메타",
  "isbn": "978-89-5555-123-4",
  "publisher": "정보출판사",
  "published_date": "2024-12-01",
  "status": "available",
  "created_at": "2025-01-15T10:30:00Z"
}

💡 상태 코드: 성공 시 201 Created 반환

Location 헤더: 생성된 자원의 URI 제공

PUT

데이터 전체 수정 (Update/Replace)

용도: 기존 데이터를 완전히 교체합니다.

특징: 안전하지 않음, 멱등성 있음

실제 예시: 도서 정보 전체 수정

📤 요청
PUT /api/books/3 HTTP/1.1
Host: library.example.com
Content-Type: application/json
Authorization: Bearer abc123token

{
  "title": "메타데이터의 이해 (개정판)",
  "author": "박메타",
  "isbn": "978-89-5555-123-4",
  "publisher": "정보출판사",
  "published_date": "2025-01-01",
  "edition": "2nd"
}
📥 응답
HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 3,
  "title": "메타데이터의 이해 (개정판)",
  "author": "박메타",
  "isbn": "978-89-5555-123-4",
  "publisher": "정보출판사",
  "published_date": "2025-01-01",
  "edition": "2nd",
  "status": "available",
  "updated_at": "2025-01-15T11:00:00Z"
}

💡 멱등성: 같은 요청을 여러 번 보내도 결과가 동일

⚠️ 주의: 모든 필드를 포함해야 함 (누락된 필드는 삭제될 수 있음)

PATCH

데이터 부분 수정 (Partial Update)

용도: 기존 데이터의 일부만 수정합니다.

특징: 안전하지 않음, 멱등성 있음

실제 예시: 도서 상태만 변경

📤 요청
PATCH /api/books/3 HTTP/1.1
Host: library.example.com
Content-Type: application/json
Authorization: Bearer abc123token

{
  "status": "checked_out"
}
📥 응답
HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 3,
  "title": "메타데이터의 이해 (개정판)",
  "author": "박메타",
  "isbn": "978-89-5555-123-4",
  "publisher": "정보출판사",
  "published_date": "2025-01-01",
  "edition": "2nd",
  "status": "checked_out",
  "updated_at": "2025-01-15T14:30:00Z"
}

💡 PUT vs PATCH:

  • PUT: 전체 데이터 교체 → 모든 필드 필요
  • PATCH: 부분 수정 → 변경할 필드만 전송
DELETE

데이터 삭제 (Delete)

용도: 서버에서 데이터를 삭제합니다.

특징: 안전하지 않음, 멱등성 있음

실제 예시: 도서 삭제

📤 요청
DELETE /api/books/3 HTTP/1.1
Host: library.example.com
Authorization: Bearer abc123token
📥 응답
HTTP/1.1 204 No Content

💡 상태 코드:

  • 204 No Content: 삭제 성공, 응답 본문 없음
  • 200 OK: 삭제 성공, 삭제된 자원 정보 반환

멱등성: 삭제된 자원을 다시 삭제해도 결과는 동일 (없음)

📊 HTTP 메서드 비교표

메서드 용도 안전성 멱등성 캐시 가능 요청 본문
GET 조회 ✅ 안전 ✅ 있음 ✅ 가능 ❌ 없음
POST 생성 ❌ 불안전 ❌ 없음 대부분 불가 ✅ 있음
PUT 전체 수정 ❌ 불안전 ✅ 있음 ❌ 불가 ✅ 있음
PATCH 부분 수정 ❌ 불안전 ✅ 있음 ❌ 불가 ✅ 있음
DELETE 삭제 ❌ 불안전 ✅ 있음 ❌ 불가 대부분 없음

4️⃣ HTTP 상태 코드: 서버의 응답 이해하기

HTTP 상태 코드는 서버가 요청을 어떻게 처리했는지 알려주는 3자리 숫자입니다.

2xx: 성공 (Success)

200 OK

요청이 성공적으로 처리됨

사용 시점: GET, PUT, PATCH 성공

GET /api/books/1 → 200 OK

201 Created

새로운 자원이 생성됨

사용 시점: POST로 새 데이터 생성 성공

POST /api/books → 201 Created
Location: /api/books/42

204 No Content

요청은 성공했지만 반환할 내용 없음

사용 시점: DELETE 성공

DELETE /api/books/42 → 204 No Content

3xx: 리다이렉션 (Redirection)

301 Moved Permanently

자원이 영구적으로 이동됨

GET /api/old-books → 301
Location: /api/books

304 Not Modified

캐시된 버전 사용 가능

GET /api/books
If-None-Match: "abc123"
→ 304 Not Modified

4xx: 클라이언트 오류 (Client Error)

400 Bad Request

잘못된 요청 형식

원인: JSON 형식 오류, 필수 필드 누락

POST /api/books
{ "titel": "오타!" }  ← title이 아님
→ 400 Bad Request
{
  "error": "title 필드는 필수입니다"
}

401 Unauthorized

인증 필요

원인: 인증 토큰 없음 또는 만료

GET /api/books
→ 401 Unauthorized
{
  "error": "인증이 필요합니다"
}

403 Forbidden

권한 없음

원인: 인증은 되었지만 권한 부족

DELETE /api/books/1
→ 403 Forbidden
{
  "error": "관리자 권한이 필요합니다"
}

404 Not Found

자원을 찾을 수 없음

원인: 존재하지 않는 자원 요청

GET /api/books/999
→ 404 Not Found
{
  "error": "도서를 찾을 수 없습니다"
}

409 Conflict

자원 충돌

원인: 중복된 데이터, 상태 충돌

POST /api/books
{ "isbn": "이미-존재함" }
→ 409 Conflict
{
  "error": "해당 ISBN이 이미 등록되어 있습니다"
}

429 Too Many Requests

요청 횟수 초과

원인: Rate limit 초과

GET /api/books
→ 429 Too Many Requests
Retry-After: 3600
{
  "error": "시간당 요청 한도 초과"
}

5xx: 서버 오류 (Server Error)

500 Internal Server Error

서버 내부 오류

원인: 서버 코드 오류, DB 연결 실패

GET /api/books/1
→ 500 Internal Server Error
{
  "error": "서버 오류가 발생했습니다"
}

503 Service Unavailable

서비스 일시 중단

원인: 서버 점검, 과부하

GET /api/books
→ 503 Service Unavailable
Retry-After: 3600

🔄 상태 코드 선택 가이드

요청이 성공했나요?
✅ 예
새 자원을 만들었나요?
✅ 예 → 201 Created
❌ 아니오 → 200 OK 또는 204 No Content
❌ 아니오
누구 잘못인가요?
클라이언트 → 4xx
서버 → 5xx

5️⃣ URI 설계 원칙: 좋은 API 만들기

1️⃣ 명사를 사용하고, 복수형을 선호

✅ /api/books

✅ /api/users

❌ /api/book (단수형)

❌ /api/getBooks (동사 사용)

2️⃣ 계층 구조를 표현

✅ /api/books/123/reviews → 도서 123번의 리뷰 목록

✅ /api/books/123/reviews/456 → 도서 123번의 리뷰 456번

✅ /api/authors/789/books → 저자 789번의 도서 목록

3️⃣ 소문자와 하이픈 사용

✅ /api/book-categories

✅ /api/user-profiles

❌ /api/bookCategories (카멜케이스)

❌ /api/book_categories (언더스코어)

4️⃣ 파일 확장자를 사용하지 않음

✅ /api/books/123

❌ /api/books/123.json

대신 Accept 헤더 사용: Accept: application/json

5️⃣ 쿼리 파라미터로 필터링, 정렬, 페이징

✅ /api/books?category=fiction&sort=title&page=2&limit=20

✅ /api/books?author=김도서&status=available

❌ /api/books/fiction/sorted-by-title/page-2

7️⃣ 실습: REST API 테스트하기

직접 REST API를 테스트해보면서 학습한 내용을 확인해봅시다!

🛠️ 도구: cURL (커맨드라인)

터미널에서 바로 API를 테스트할 수 있는 도구입니다.

1. GET 요청 보내기

curl https://api.github.com/users/octocat

GitHub API로 사용자 정보 조회

2. 헤더 포함하여 요청

curl -H "Accept: application/json" \
     https://api.github.com/users/octocat

Accept 헤더로 JSON 형식 요청

3. POST 요청 보내기

curl -X POST \
     -H "Content-Type: application/json" \
     -d '{"title":"New Book","author":"Kim"}' \
     https://api.example.com/books

JSON 데이터를 포함한 POST 요청

4. 인증 토큰 포함

curl -H "Authorization: Bearer YOUR_TOKEN" \
     https://api.example.com/protected-resource

Bearer 토큰으로 인증

🌐 도구: 브라우저 개발자 도구

웹 브라우저의 개발자 도구(F12)로 네트워크 요청을 확인할 수 있습니다.

1단계: F12 키를 눌러 개발자 도구 열기
2단계: Network 탭 선택
3단계: 웹페이지에서 액션 수행 (클릭, 검색 등)
4단계: API 요청 확인 (Request Headers, Response 등)

📮 도구: Postman

GUI 기반 API 테스트 도구로, 가장 널리 사용됩니다.

다운로드: https://www.postman.com/downloads/

주요 기능

  • 직관적인 GUI로 요청 작성

8️⃣ 보안과 인증

REST API는 공개된 인터페이스이므로 보안이 매우 중요합니다.

1️⃣ API 키 (API Key)

사용 방법: 요청 헤더 또는 쿼리 파라미터에 포함

헤더 방식

GET /api/data HTTP/1.1
X-API-Key: abc123xyz789

쿼리 파라미터 방식

GET /api/data?api_key=abc123xyz789 HTTP/1.1

✅ 장점: 간단하고 구현 쉬움

❌ 단점: 키가 노출되면 보안 위험, 사용자별 권한 관리 어려움

2️⃣ OAuth 2.0

사용 방법: 토큰 기반 인증, 가장 널리 사용되는 방식

인증 흐름

1. 클라이언트: 인증 요청
2. 인증 서버: 사용자 로그인 페이지 표시
3. 사용자: 로그인 및 권한 승인
4. 인증 서버: Access Token 발급
5. 클라이언트: Access Token으로 API 요청

토큰 사용 예시

GET /api/user/profile HTTP/1.1
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

✅ 장점: 안전하고 표준화됨, 세밀한 권한 제어 가능

❌ 단점: 구현 복잡도 높음

3️⃣ JWT (JSON Web Token)

사용 방법: 자체 포함형 토큰, 서버 상태 저장 불필요

JWT 구조

Header
{"alg": "HS256", "typ": "JWT"}
.
Payload
{"userId": 123, "role": "admin"}
.
Signature
암호화된 서명

실제 JWT

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.
eyJ1c2VySWQiOjEyMywicm9sZSI6ImFkbWluIn0.
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

✅ 장점: 무상태성, 서버 부하 감소, 다양한 시스템 간 공유 가능

❌ 단점: 토큰 크기가 큼, 토큰 무효화 어려움

🔒 보안 Best Practices

  • ✅ HTTPS 사용 필수: 모든 API 통신은 암호화
  • ✅ Rate Limiting: API 남용 방지
  • ✅ 입력 검증: SQL Injection, XSS 방지
  • ✅ CORS 설정: 허용된 도메인만 접근 가능
  • ✅ 민감 정보 제외: 비밀번호, 개인정보는 로그에 남기지 않음
  • ✅ 버전 관리: API 버전을 명시 (/api/v1/)

💡 보안 용어 쉽게 이해하기

Rate Limiting (사용량 제한)

📚 도서관 비유:

도서관에서 한 사람이 책을 하루에 10권까지만 대출할 수 있게 제한하는 것처럼, API도 일정 시간 동안 요청 횟수를 제한합니다.

예시:

  • 한 사용자가 1분에 60번까지만 API를 호출할 수 있음
  • 초과하면 429 Too Many Requests 에러 응답

왜 필요한가요?

  • 🛡️ 서버 보호: 악의적인 공격(DoS)이나 실수로 너무 많은 요청을 막아요
  • ⚖️ 공평한 사용: 한 사람이 서버 자원을 독점하지 못하게 해요
  • 💰 비용 관리: 클라우드 서비스는 요청 횟수로 과금되기도 해요

실제 응답 예시:

HTTP/1.1 429 Too Many Requests
X-RateLimit-Limit: 60          ← 1분에 60번까지 가능
X-RateLimit-Remaining: 0       ← 남은 횟수: 0
X-RateLimit-Reset: 1640000000  ← 다시 가능한 시각

{
  "error": "요청 한도를 초과했습니다. 60초 후 다시 시도하세요."
}
CORS (Cross-Origin Resource Sharing)

📚 도서관 비유:

우리 도서관 회원증으로는 우리 도서관 자료만 이용할 수 있고, 다른 도서관 자료는 이용할 수 없는 것처럼, 웹사이트도 "어느 웹사이트에서 내 API를 호출할 수 있는지" 제한합니다.

상황 설명:

  • 출처(Origin): 도메인 + 포트 조합

    예: https://library.com:443

  • 같은 출처: library.com → library.com ✅ OK
  • 다른 출처: evil-site.com → library.com ❌ 차단!

왜 필요한가요?

악의적인 웹사이트가 사용자 모르게 여러분의 로그인 정보를 훔쳐서 은행 API나 도서관 API를 몰래 호출하는 것을 막기 위해서입니다.

서버 설정 예시:

# 응답 헤더에 허용할 도메인 명시
Access-Control-Allow-Origin: https://library.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization

# 모든 도메인 허용 (공개 API의 경우)
Access-Control-Allow-Origin: *

→ 지정된 도메인만 API를 호출할 수 있게 됩니다.
